# Palova — Instructions pour Claude

Application de réservation de terrains de padel. Backend API séparé du frontend Next.js.

## Démarrage

```bash
# 1. Démarrer PostgreSQL + Redis (Docker)
"C:\Program Files\Docker\Docker\resources\bin\docker-compose-v1.exe" up -d

# 2. Backend (dans un terminal, dossier backend/)
npm run dev
# → http://localhost:3001

# 3. Frontend (dans un autre terminal, dossier frontend/)
npm run dev
# → http://localhost:3000
```

> **IMPORTANT — Docker** : La version installée (20.10.23) n'a pas le plugin `compose`. Toujours utiliser `docker-compose-v1.exe`, jamais `docker compose`.

## Logique métier clé

**Zéro double-réservation** = deux niveaux :
1. **Redis SET NX** (`lock:court:{id}:{startTime}`, TTL 10 min) — bloque dès le début du checkout
2. **PostgreSQL Serializable + SELECT FOR UPDATE** — garantie absolue à la confirmation

**SSE (temps réel)** — `SSEService` singleton broadcast aux clients connectés sur un terrain :
- `slot_held` → créneau pris (PENDING)
- `slot_confirmed` → confirmé
- `slot_released` → libéré (annulation ou expiration)

**Créneaux** : 8h–22h heure Paris (UTC+2 hardcodé), pas de 30 min.

**Perfs « rush de minuit » (2026-07-18)** — durcissement du chemin chaud de lecture (audit perfs : des centaines de joueurs rafraîchissent la grille Réserver à l'ouverture des créneaux). Quatre briques, backend + Caddy, AUCUN changement frontend :
1. **Micro-cache disponibilités** `src/services/availabilityCache.ts` (mémoire process, TTL 2 s + single-flight, garde-fou 1000 entrées) servi par `AvailabilityService.getClubAvailabilityBySlug` — la route publique `GET /:slug/availability` ne touche plus la base sur un hit (la résolution slug→club vit DANS le calcul caché). ⚠️ **Invariant : toute écriture qui change une disponibilité DOIT appeler `invalidateClubAvailability(clubId)`** — câblé sur hold, confirm, performCancel, admin create/reschedule, séries (create/cancel) et le job cleanup (`releaseExpiredHolds`, extrait et testé) ; une invalidation oubliée se répare seule à l'expiration du TTL. Cache **désactivé sous jest** (TTL 0 — les suites de routes mockent Prisma requête par requête) ; les tests du cache le réactivent via `_setAvailabilityCacheTtl`.
2. **Fini le N+1 d'availability** : `getClubAvailability` = 3 requêtes SQL constantes (club, terrains, réservations groupées `resourceId IN`) au lieu de 2+2×terrains ; `classifySlot` calculé UNE fois par créneau (le prix en découle — `slotPriceCents` refaisait la même marche luxon en interne). Grille construite par le helper pur partagé `buildSlots`.
3. **Cache d'identité auth** `src/middleware/authCache.ts` (TTL 30 s, désactivé sous jest) : `tokenRevoked` ne fait plus un SELECT user par requête authentifiée. ⚠️ **Invariant : toute écriture révocante (tokenVersion++, deletedAt) DOIT appeler `invalidateAuthIdentity(userId)`** — câblé sur reset-password et suppression de compte (sans quoi le NOUVEAU token post-reset serait refusé jusqu'à 30 s).
4. **Compression Caddy** : snippet `(compress)` (`encode zstd gzip`) importé par les 3 blocs du `Caddyfile`, liste de types explicite **sans `text/event-stream`** (les 4 canaux SSE traversent sans compresseur). Ni Express ni l'API n'en faisaient nulle part.
Les deux caches sont **process-local** (comme les canaux SSE) : passage en multi-instance un jour ⇒ les remplacer par Redis, même chantier que le pub/sub SSE. Tests : `availabilityCache`/`authCache` (nouvelles suites), `availability.service` (N+1/groupage), `clubs.availability.routes` (nouvelle — 404/400/câblage cache), blocs « invalide le cache » dans `reservation.service`/`reservation.series`/`cleanup.job`/`auth.routes`/`account.service`.

## Bugs connus et résolus

- **EPIPE crash** dev server → géré dans `frontend/instrumentation.ts`
- **Hydration mismatch** extension ColorZilla → `suppressHydrationWarning` sur `<html>` et `<body>` dans layout.tsx
- **Page vide au clic** sur un terrain → `loading` initialisé à `true` dans courts/[id]/page.tsx
- **Slot count** : 8h–22h / 30min step / 60min durée = **27 créneaux** (pas 28)
- **Backend `npm run dev` → `[nodemon] app crashed` (Windows)** alors que le serveur tourne en réalité (`/health` répond) : nodemon exécutait `ts-node` (shim `ts-node.cmd`) et **perdait la trace du process node** → faux « app crashed » + auto-reload cassé + node orphelin qui garde le port 3001 (EADDRINUSE au reload). Corrigé en lançant node directement : `--exec "node -r ts-node/register src/app.ts"` (mêmes deps, pas de shim).
- **Login sur l'hôte plateforme `localhost:3000` → « on reste sur un écran de login, pas de redirection » (DEV UNIQUEMENT, 2026-07-15)** : un compte **staff** (OWNER/ADMIN/STAFF) qui se connecte sur la plateforme est redirigé vers l'admin du club **par changement de sous-domaine** (`window.location.assign(clubUrl(slug,'/admin'))` dans `finishAuth`, `lib/postAuth.ts`). Or **Chrome refuse tout cookie partagé entre `*.localhost`** (vérifié : `domain=.localhost`/`domain=localhost` sont normalisés en host-only `localhost`, invisibles sur `padel-arena-paris.localhost`) → le cookie `token` ne suit pas le hop → le proxy du sous-domaine club, ne voyant pas de token, renvoie sur `/login`. **Prod inchangée** (`.palova.fr` couvre `*.palova.fr`). Corrigé par un **pont de session dev-only** : route publique **`/session-bridge`** (`app/session-bridge/page.tsx`, ajoutée à `PUBLIC_PATHS`) — `finishAuth` route le staff via `sessionBridgeUrl()` **seulement si `sessionCookieIsHostOnly(host)`** (`lib/session.ts`, miroir de `cookieDomainAttr` : host-only ⇔ root absent ou `localhost`), passant le token dans le **fragment `#`** (jamais envoyé au serveur/journaux) ; le pont repose le cookie côté sous-domaine club puis `window.location.replace('/admin')`. Le pont **n'agit que sur un hôte host-only** → en prod il ignore le fragment (`→ /`, pas de fixation de session) et n'est jamais emprunté. Navigation plein-écran isolée dans `lib/nav.ts` (`hardNavigate`/`hardReplace`/`currentHost`) pour la testabilité (jsdom verrouille `window.location`). Le **super-admin** (`router.push('/superadmin')`) et le **joueur simple** (`router.push('/clubs')`) sont same-origin → jamais concernés. Tests : `__tests__/postAuth.test.ts` (pont en dev / direct en prod).
- **« Tout Chrome rame en local, toutes les pages en Chargement… » (DEV, 2026-07-15)** : Chrome plafonne à **6 connexions HTTP/1.1 par origine** (tous onglets confondus) vers `localhost:3001`, or chaque page connectée ouvrait **3+ `EventSource` identiques** sur le flux SSE de la cloche (`ClubNav` ×2 + `NotificationBell`, +1 sur /parties et Messages) **jamais fermés** → 2 onglets Palova suffisaient à saturer le quota et **toutes** les requêtes API de tous les onglets faisaient la queue indéfiniment (serveur parfaitement sain pendant ce temps — curl à 6-64 ms). Invisible en prod (HTTPS/Caddy → HTTP/2 multiplexé, pas de limite de 6). Corrigé par le **singleton `lib/notificationsStream.ts`** (`subscribeNotifications(token, cb)` → unsubscribe) : UNE EventSource par onglet partagée par tous les abonnés, fermée au départ du dernier, remplacée au changement de token ; **pas de `onerror → close()`** (l'ancien pattern tuait le live au premier pépin réseau — on laisse la reconnexion native d'EventSource). Consommateurs migrés : `ClubNav` ×2, `NotificationBell`, `OpenMatches`, `MessagesHub` (les streams contextuels par-conversation/par-chat/par-court restent des EventSource directs). Vérifié CDP : /parties connecté = **1 flux vivant** (4 avant). Diagnostic type : `Get-NetTCPConnection -RemotePort 3001 -State Established` bloqué à 6, toutes tenues par chrome. Tests : `__tests__/notificationsStream.test.ts`.
- **`start.ps1` → « ECHEC : le backend ne repond pas » après une mise en veille** : la pile dev de la veille survit au sommeil en **zombie** (process backend vivant mais socket :3001 morte → il échappe au kill par port `Stop-PortOwner`) et son `cmd` garde **`logs\backend.log` verrouillé** → la redirection `> backend.log` du nouveau `cmd /c npm run dev` échoue à l'ouverture et le nouveau backend meurt **sans écrire une ligne**. Corrigé (2026-07-05) : `Stop-StaleStack` purge la pile **par identité** (node/cmd dont la CommandLine référence `backend\`/`frontend\`/`logs\` du repo **ou** `ts-node/register src/app.ts` — le child ts-node n'a pas le chemin du repo dans sa ligne de commande) + attente 2 s (relâche des handles) + repli **log horodaté** dans `Start-Hidden` si le log reste verrouillé (plus jamais d'échec muet). Diagnostic type : `backend.log` qui se termine par une rafale de `[NODE-CRON] missed execution` = process gelé/machine en veille, pas un crash.

## Prix promotionnels sur les terrains (v1) ✅ implémenté

Le club programme des remises datées sur ses terrains : **`Promotion`** (`backend/src/services/promotion.service.ts`, migration additive `add_promotions`) — nom, période `startDate`/`endDate` (jours calendaires inclus, comparaison lexicale `YYYY-MM-DD`), remise **`PERCENT`** (1–100 %) ou **`FIXED`** (prix fixe en euros), **fenêtre horaire optionnelle** (`windowStart`/`windowEnd` en minutes depuis minuit, créneau à cheval sur minuit hors périmètre), ciblage **`PromotionResource`** (table de jointure ; vide = tous les terrains du club). `PromotionService` (CRUD + `validate` privée réutilisée création/màj, `updatePromotion` en transaction pour la resynchro des ressources ciblées) + **`loadActivePromotions(clubId, localDate)`** (filtre `enabled` + période, une requête par grille).

**Calcul de prix** — nouveau **`effectiveSlotPriceCents(baseCents, promos, resourceId, start, end, tz)`** dans `pricing.ts` (le client gagne : retient le prix le plus bas parmi les promos éligibles, une promo ne fait **jamais** monter le prix). Branché à deux endroits : (1) **`availability.service.ts`** — `buildSlots` (méthode privée pure, cœur de la réécriture perf batchée « rush de minuit ») reçoit les promos déjà chargées une fois par grille et n'appelle `effectiveSlotPriceCents` **que si `promos.length > 0`** (zéro surcoût CPU sur un club sans promo active) ; la grille expose **`TimeSlot.originalPrice`/`promoName`** quand un créneau est remisé. (2) **`reservation.service.ts`** — `holdSlot` charge les promos **en direct** (`loadActivePromotions`) et fige le prix remisé dans `totalPrice` au moment du hold — le montant réellement facturé est donc toujours calculé à la volée, **jamais lu depuis le cache de disponibilité** (qui peut rester périmé jusqu'à 2 s, purement cosmétique côté affichage).

Admin **`/admin/promotions`** (nav « Promotions », icône `euro`, gating `isClubAdmin`) : vitrine + studio (`PromotionCard`/`PromotionForm`), CRUD via `POST/PATCH/DELETE …/admin/promotions[/:id]` (`requireClubMember('ADMIN')`). Front : `lib/adminPromotions.ts` (`promoStatus` upcoming/running/past, `groupPromotions`, `discountLabel`, `windowLabel`, tous purs).

**Écarts connus, à trancher avant d'aller plus loin (non bloquants, documentés à la relecture)** : `rescheduleReservation`/`adminRescheduleReservation` **n'appliquent pas** les promos (seul `holdSlot` le fait — déplacer une résa dans/hors d'une fenêtre promo recalcule au prix de base) ; le cache de disponibilité **n'est pas invalidé** à la création/modification/suppression d'une promo (le prix *affiché* peut rester périmé jusqu'à expiration du TTL, le prix *facturé* est toujours correct). Migration additive uniquement. Tests : `pricing`/`availability.service`/`reservation.service`/`promotion.service`/`admin.promotions.routes` (backend) ; `adminPromotions`/`AdminPromotions`/`AdminLayout` (frontend).
