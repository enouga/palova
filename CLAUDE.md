# Palova — Instructions pour Claude

Application de réservation de terrains de padel. Backend API séparé du frontend Next.js.

## Démarrage

```bash
# 1. Démarrer PostgreSQL + Redis (Docker)
"C:\Program Files\Docker\Docker\resources\bin\docker-compose-v1.exe" up -d

# 2. Backend (dans un terminal, dossier backend/)
npm run dev
# → http://localhost:3001

# 3. Frontend (dans un autre terminal, dossier frontend/)
npm run dev
# → http://localhost:3000
```

> **IMPORTANT — Docker** : La version installée (20.10.23) n'a pas le plugin `compose`. Toujours utiliser `docker-compose-v1.exe`, jamais `docker compose`.

Incompatible avec Turbopack. Ne pas l'activer dans next.config.ts.

## Logique métier clé

**Zéro double-réservation** = deux niveaux :
1. **Redis SET NX** (`lock:court:{id}:{startTime}`, TTL 10 min) — bloque dès le début du checkout
2. **PostgreSQL Serializable + SELECT FOR UPDATE** — garantie absolue à la confirmation

**SSE (temps réel)** — `SSEService` singleton broadcast aux clients connectés sur un terrain :
- `slot_held` → créneau pris (PENDING)
- `slot_confirmed` → confirmé
- `slot_released` → libéré (annulation ou expiration)

**Créneaux** : 8h–22h heure Paris (UTC+2 hardcodé), pas de 30 min.

## Bugs connus et résolus

- **EPIPE crash** dev server → géré dans `frontend/instrumentation.ts`
- **Hydration mismatch** extension ColorZilla → `suppressHydrationWarning` sur `<html>` et `<body>` dans layout.tsx
- **Page vide au clic** sur un terrain → `loading` initialisé à `true` dans courts/[id]/page.tsx
- **Slot count** : 8h–22h / 30min step / 60min durée = **27 créneaux** (pas 28)

## Inscription par email + code (v1) ✅ implémenté

L'inscription se valide par **code reçu par email**. `POST /api/auth/register` crée un compte **non vérifié** (`User.emailVerified`, migration `add_email_verification` + backfill des comptes existants à `true`) et envoie un code à 6 chiffres (modèle `EmailVerification` : code **hashé bcrypt**, expiry 15 min, 5 essais max, cooldown renvoi 60 s). `POST /api/auth/verify-email` valide le code → renvoie le JWT ; `POST /api/auth/resend-code` renvoie un code (réponse neutre, pas d'énumération). `login` renvoie **403 `EMAIL_NOT_VERIFIED`** pour un compte non vérifié. Le service capte l'argent du joueur via **`Reservation.userId` (organisateur) ET `ReservationParticipant.userId`** (scope `resource.clubId`, Reservation n'a pas de `clubId`), net des `refundedAmount`, **`MONEY_METHODS`** seuls comptent (miroir `accounting.service.ts`) ; `PACK_CREDIT`/`WALLET`/`MEMBER` = prépayé/abo affichés à part ; buckets mois/heatmap **tz-aware** (Luxon + `Club.timezone`) ; reste dû via `slotPriceCents` quand `totalPrice = 0`. Heuristiques documentées : **no-show** = résa CONFIRMED passée, facturable, jamais réglée dont le joueur est organisateur (libellé « estimation ») ; **annulation tardive** dérivée de `cancelledAt` vs `Club.cancellationCutoffHours` (pas de motif stocké). Frontend : page client (`useParams`, pas de `new Date()` au rendu — les relatifs sont calculés serveur), graphiques **SVG faits main** `frontend/components/admin/stats/{MonthlyRevenueChart,DayHourHeatmap,PaymentMethodChart}.tsx`, **toute la math/les libellés purs** dans `frontend/lib/memberStats.ts` (testés), montants via `fmtEuros`/`toCents`. Types `MemberHistory` dans `lib/api.ts` + `adminGetMemberHistory`. Tests : `backend/src/services/__tests__/memberStats.service.test.ts`, `frontend/__tests__/{memberStats,MemberHistory,AdminMembersNav}.test.tsx`. Plan : `~/.claude/plans/moonlit-enchanting-dream.md`.

## Paiement en ligne des inscriptions (tournois & events) (v1) ✅ implémenté

Une épreuve peut exiger un **règlement CB en ligne** à l'inscription, **au choix du club et par épreuve** (case « Inscription à régler en ligne » dans `/admin/tournaments` et `/admin/events`), **activable seulement si le club a activé le paiement en ligne** (compte Stripe Connect `ACTIVE`). Réutilise l'infra Stripe Connect des réservations. Migration additive **`add_registration_online_payment`** : `Tournament.requirePrepayment`/`ClubEvent.requirePrepayment`, enum **`RegistrationPaymentStatus { NONE DUE PAID REFUNDED }`** + `paymentStatus`/`paymentDeadline` sur les deux modèles d'inscription, 2 FK nullables sur `Payment` (`tournamentRegistrationId`/`eventRegistrationId`). **⚠️ Migration NON appliquée en base** (développée en worktree isolé « types-only » via `prisma generate`) — **à générer/appliquer au merge** (`prisma migrate dev --name add_registration_online_payment`). **Tenue de la place pendant le paiement** : l'inscription payante naît `CONFIRMED + paymentStatus DUE + paymentDeadline = now+15 min` (helpers purs `backend/src/services/registrationPayment.ts` : `REGISTRATION_HOLD_MINUTES`, `holdDeadline`, `occupiesSpotWhere` — une DUE expirée ne compte pas dans la capacité — , `entryFeeCents`) ; le paiement la passe `PAID` (`confirmRegistrationPayment`, flip conditionnel `updateMany DUE→PAID` idempotent + `Payment` ONLINE/CAPTURED avec `receiptNo`, déclenché par le **client ET le webhook** `payment_intent.succeeded` routé sur la metadata). **Liste d'attente gratuite** : carte enregistrée via **SetupIntent** (webhook `setup_intent.succeeded`), débit **off-session** à la promotion (`chargePromotedRegistration`, hors transaction, `idempotencyKey`) ; **carte refusée → on libère et on promeut le suivant** (récursif) ; charge **best-effort** (`safeCharge`, ne fait jamais échouer une annulation). **Tournoi = le capitaine paie l'`entryFee` de l'équipe**. **Remboursement auto** à la désinscription **avant clôture** (`RefundService`, best-effort → `paymentStatus REFUNDED`). Le **cleanup job** libère les `CONFIRMED+DUE` expirées chaque minute (early-return rendu local) + promeut le suivant ; les `WAITLISTED+DUE` (deadline null) ne sont jamais touchées. `StripeService` : `createRegistration{Payment,Setup}Intent`, `chargeRegistrationOffSession(…, idempotencyKey?)`. Routes : `POST /api/{tournaments,events}/:id/registrations/:regId/{intent,confirm-payment}` (ownership 403, `NOT_PAYABLE` si non DUE) ; `register` renvoie `{ registration, payment: { mode:'payment'|'setup' } | null }` ; garde-fou admin `ONLINE_PAYMENT_NOT_ENABLED` (409, Stripe non actif ou montant < 0,50 €). Front : `StripePaymentStep` rendu **target-agnostic** (callbacks `createIntent`/`confirm`, partagé avec `BookingModal`), parcours paiement/card-setup sur `/tournois/[id]` et `/events/[id]` (flux **gratuit strictement inchangé**), `api.createRegistrationIntent`/`confirmRegistrationPayment`. **Épreuve gratuite = comportement actuel intact.** Hors v1 : partage entre coéquipiers, relances/délai de paiement waitlist, remboursement partiel auto, règlement via caisse/porte-monnaie, e-mail dédié « carte refusée » (réutilise l'e-mail de désinscription). Spec & plan : `docs/superpowers/{specs,plans}/2026-06-25-paiement-en-ligne-inscriptions*`.
